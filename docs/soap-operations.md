## Call strategy decided from this schema

**Sales history (for DRR):**
1. `SearchSaleOrder` with `Channel` (optional, repeat once per channel of interest or omit to get all), `FromDate`/`ToDate` set to the DRR lookback window, and required `SearchOptions.DisplayStart` (+ `DisplayLength`) for pagination. Response only returns order-level summary fields (`Code`, `Channel`, `Status`, dates) — **no line items, no SKU, no facility**.
2. For each returned order `Code`, call `GetSaleOrder` to get `SaleOrderItems[]` with `ItemSKU` and `FacilityCode`. `Channel` also appears on the order level in this response.
3. **Important:** `SaleOrderItem` has no `Quantity` field. Unicommerce models each unit as its own `SaleOrderItem` record, so **units sold for a SKU = count of `SaleOrderItem` rows with that `ItemSKU`**, not a summed quantity field. `drrEngine.js` must count rows, not sum a quantity attribute.
4. This means call volume is `1 SearchSaleOrder page + N GetSaleOrder calls` (N = orders in the window) — acceptable for a once-daily batch job at this business's order volume, but `salesPuller.js` should log progress and consider only fetching `Status` values that represent completed/dispatched sales (filter via the `Status` field on `SearchSaleOrderRequest`) to avoid pulling cancelled orders into DRR.

**Inventory snapshot (for current stock):**
- Use `GetBulkItemTypeInventory` (NOT the single-SKU `GetItemTypeInventory`) — takes `SkuCodes[]` + `FacilityCodes[]` in one call, returns `InventoryDetails.ItemInventory[].Facilities.FacilityInventory[]` with `{FacilityCode, FacilityName, Inventory}` per SKU.
- **Undocumented limit confirmed live: max 50 SKU codes per call** (`"skuCodes Maximum 50 sku codes are allowed"`, error code 1001/`MISSING_REQUIRED_PARAMETERS`) — exceeding it returns `Successful: false` with **no `InventoryDetails` at all** (not a partial result). `inventoryPuller.js` batches SKUs into chunks of 50 and merges results across calls.
- Facility codes passed in the request are echoed back uppercased in an `Errors` block (`INVALID_FACILITY_CODE`) even when using the correct casing — but `InventoryDetails` still comes back populated and correct in that case. This looks like a harmless validation-logging quirk on Unicommerce's side, not a real failure; `pullInventorySnapshot` doesn't treat `Successful: false` as fatal, it just reads whatever `InventoryDetails` is present.
- `GetInventorySnapshot` was also inspected as an alternative — it gives a richer per-SKU breakdown (`Inventory`, `OpenSale`, `OpenPurchase`, `InventoryBlocked`, `PutawayPending`, etc.) but is **not facility-split** in its response, so it's less useful for a facility-aware days-of-cover calc. Kept as a documented option but not used.

**Facilities are not a fixed, known-in-advance set.** The business operates at least two facilities seen live (`SBAB_Holisol_Kolkata` and `GC_Holisol`), not just the one visible in the manually-downloaded ledger CSV. `run.js` derives the facility list dynamically from `FacilityCode` values found in the pulled sales rows each run, rather than relying on a hardcoded config list.
